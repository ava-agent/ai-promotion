I Built a Travel Planning AI. It Failed 47 Times Before It Worked.

Three months ago, I started building an AI travel agent. I thought it would be simple: take user preferences, search for attractions, generate an itinerary. How hard could it be?

47 failed iterations later, I finally understand why travel planning is one of the hardest problems for AI agents.

---

## The Naive Approach (Iterations 1-5)

My first version was embarrassingly simple:

1. User inputs destination + dates
2. LLM generates a list of attractions
3. Agent schedules them across days
4. Output a pretty itinerary

It worked great for my test case: "3 days in Tokyo."

Then a user tried: "Family trip to Paris, 2 adults, 2 kids (ages 4 and 7), one uses a wheelchair."

**Everything broke.**

The agent suggested:
- The Eiffel Tower (no wheelchair access info)
- A 3-hour walking tour of Montmartre (impossible with a 4-year-old)
- A romantic dinner cruise (not exactly kid-friendly)

I realized: travel planning isn't about attractions. It's about constraints, preferences, and context that humans handle intuitively but AI struggles to understand.

---

## The Context Problem (Iterations 6-15)

I added a "user profile" system. Users could specify:
- Mobility limitations
- Dietary restrictions
- Budget range
- Travel style (adventure, relaxation, culture)

This helped, but created a new problem: users don't know what they want until they see options.

**Example conversation:**
```
User: "I want a cultural trip to Rome."
Agent: Generates 5 museums, 3 churches, 2 historical sites
User: "This looks boring. Can we add some fun stuff?"
Agent: Confused - you said cultural
```

The issue: users express preferences imprecisely. "Cultural" could mean museums, or it could mean "authentic local experiences," or "not tourist traps."

**Solution:** I built an interactive preference discovery system. Instead of asking users to specify everything upfront, the agent proposes options and learns from reactions.

```
Agent: "For Rome, I'm thinking a mix of ancient history and modern food culture. Does that sound right?"
User: "Yes, but less ruins, more food."
Agent: Adjusts plan accordingly
```

This reduced itinerary rejections by 60%.

---

## The Real-Time Data Problem (Iterations 16-25)

Even with perfect preferences, plans fail because of bad data.

**Case study:** A user followed my agent's recommendation to a highly-rated restaurant. When they arrived, it had permanently closed 3 months ago.

**Why it happened:**
- My knowledge base was from web scraping (6-month-old data)
- No real-time verification
- LLM hallucinated current status

**The fix:** I built a multi-layer verification system:

1. **Primary data**: Official websites + booking platforms (APIs)
2. **Secondary verification**: Recent reviews + social media mentions
3. **Fallback**: User feedback loop (flag outdated info)

This required integrating 12 different APIs:
- Google Places (attractions)
- OpenTable/Resy (restaurants)
- Ticketmaster/GetYourGuide (activities)
- Weather APIs (outdoor plans)
- Flight/travel APIs (logistics)

**Cost:** $0.15-0.30 per itinerary in API calls alone.

**Lesson:** Real-time data is expensive but necessary. Free data is stale data.

## The Temporal Reasoning Problem (Iterations 26-35)

Here's a simplified itinerary my agent generated:

```
Day 1:
09:00 - Visit Colosseum (2 hours)
11:00 - Travel to Vatican (30 min)
11:30 - Vatican Museums (3 hours)
14:30 - Lunch
```

Looks reasonable, right?

**What the agent missed:**
- Colosseum tickets sell out days in advance
- Vatican Museums are closed on Sundays
- 30 minutes is optimistic for Rome traffic
- 2 PM is when Italians eat lunch (restaurants full)

The agent understood locations but not temporal constraints.

**Solution:** I built a "temporal constraint solver" that considers:

1. **Booking requirements**: Which attractions need advance tickets?
2. **Operating hours**: Seasonal variations, holiday closures
3. **Crowd patterns**: Peak times to avoid
4. **Travel time buffers**: Realistic transit estimates
5. **Meal timing**: Local dining customs

This increased itinerary feasibility from 40% to 85%.

---

## The Optimization Problem (Iterations 36-42)

By iteration 36, I had:
- Accurate preferences
- Real-time data
- Temporal constraints

But the itineraries were... boring.

**User feedback:** "This feels like a checklist, not a vacation."

**The issue:** I was optimizing for efficiency (minimize travel time, maximize attractions) instead of experience quality.

**What I learned:** Travel isn't a logistics problem. It's an experience design problem.

**Changes I made:**

1. **Pacing variety**: Mix active and relaxed activities
2. **Discovery moments**: Leave room for serendipity
3. **Energy mapping**: Match activities to energy levels (morning = active, afternoon = relaxed)
4. **Buffer time**: 20% unplanned time for exploration
5. **Theme days**: Instead of scattered activities, create daily narratives

**Example transformation:**

Before:
```
Day 1: Colosseum → Vatican → Trevi Fountain → Pantheon
```

After:
```
Day 1 - Ancient Rome Day:
Morning: Colosseum + Roman Forum (fresh energy for walking)
Lunch: Trastevere (authentic Roman neighborhood)
Afternoon: Wander + gelato (low-key exploration)
Evening: Sunset at Pincian Hill (romantic end to ancient day)
```

User satisfaction jumped from 3.2/5 to 4.4/5.

## The Multi-Agent Coordination Problem (Iterations 43-47)

The final challenge: group travel.

**Scenario:** A family of 4 with different interests:
- Dad: History buff
- Mom: Foodie
- Teen: Instagram spots
- Kid: Playground + ice cream

My single-agent approach couldn't balance competing preferences.

**Solution:** I built a multi-agent negotiation system:

1. **Preference agents**: Each traveler has an agent advocating for their interests
2. **Mediator agent**: Balances competing requests
3. **Constraint checker**: Ensures logistics work
4. **Experience synthesizer**: Creates coherent narrative

**Example negotiation:**

```
Dad's agent: "Morning at Colosseum (high priority)"
Mom's agent: "Lunch at Testaccio market (high priority)"
Teen's agent: "Afternoon at colorful neighborhoods (medium priority)"
Kid's agent: "Playground break (required)"

Mediator: "Colosseum morning → Testaccio lunch (combine with market tour) → 
Trastevere afternoon (colorful + playground nearby) → Evening gelato"
```

All agents approve. Consensus reached.

This was the breakthrough moment. The agent finally felt like a helpful travel companion, not a scheduling robot.

## What I Learned

### 1. Travel Planning is a Wicked Problem
It's not just about information retrieval. It's about:
- Understanding implicit preferences
- Balancing competing constraints
- Adapting to real-time changes
- Creating experiences, not schedules

### 2. Data Freshness is Everything
A travel agent with 6-month-old data is worse than no agent at all. Users trust recommendations, and broken trust is hard to rebuild.

### 3. Humans Don't Know What They Want
Direct questions ("What's your travel style?") get vague answers. Interactive discovery ("Rate these 5 itineraries") reveals true preferences.

### 4. Efficiency is the Wrong Metric
The best itinerary isn't the one with the most attractions. It's the one that creates the best memories. Sometimes that means doing less.

### 5. AI Needs to Be a Partner, Not a Tool
Users don't want a search engine with extra steps. They want a travel companion who understands them, anticipates their needs, and helps them discover things they didn't know they wanted.
