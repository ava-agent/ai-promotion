## 📝 内容模板

### 1. 技术洞察类 (技术观察)
**目标**: 展示技术深度，引发共鸣
**结构**:
```
[反直觉Hook]

大多数人[常规做法]。
但实际上：[核心洞察]。

[简短解释/案例]

这就是为什么[解决方案/观点]。

[互动问题]
```

**示例**:
```
做AI产品最大的坑不是算法，是数据质量。

很多人只关注模型精度，
但忽略了训练数据的偏见和分布。

我们踩过的坑：
- 训练数据全是高端用户，上线后普通用户投诉
- 时间序列数据没有考虑节假日，预测完全不准
- 标注数据有歧义，模型学到的是错误模式

数据质量 > 算法优化

你们在做AI时踩过哪些数据坑？评论区聊聊 👇
```

**检查清单**:
- [ ] 有反直觉的Hook（让人想点开）
- [ ] 具体的踩坑案例（真实可信）
- [ ] 个人经验或观察（第一人称）
- [ ] 明确的结论（观点鲜明）
- [ ] 互动问题结尾（引导评论）

### 3. 踩坑实录类 (排雷指南)
**目标**: 分享真实踩坑经历，帮助他人避坑
**结构**:
```
[血泪Hook] 😭

[问题描述]

[排查过程 - 简短]

[最终原因]

[教训总结]

[给读者的建议]
```

**示例**:
```
被Docker坑了2天，最后发现是个权限问题 😭

现象：
本地运行正常，部署到服务器就报错。

排查过程：
1. 检查代码 → 没问题
2. 检查环境变量 → 没问题
3. 检查依赖版本 → 没问题

最后发现：Docker容器里的时区不对，
导致时间戳计算错误。

解决：Dockerfile里加上 `RUN ln -sf /usr/share/zoneinfo/Asia/Shanghai /etc/localtime`

教训：
- 容器化一定要注意时区
- 日志里要打印完整的上下文

你们有类似的踩坑经历吗？
```

**检查清单**:
- [ ] 情感化的Hook（"被坑了"、"崩溃了"）
- [ ] 具体的问题现象
- [ ] 排查步骤（3步以内）

## 🎨 风格指南

### Hook写作技巧
1. **数字冲击**: "被坑了2天..."、"效率提升80%"
2. **情感共鸣**: "崩溃了"、"才发现"、"血泪教训"
3. **反常识**: "最好的代码是没有代码"
4. **直接利益**: "推荐一个效率神器"

### 语言特点
- **口语化**: "其实"、"说实话"、"说白了"
- **接地气**: "踩坑"、"排雷"、"翻车"
- **第一人称**: "我发现"、"我觉得"、"我的体会"
- **emoji点缀**: 🚀💡😭👇（适度使用）

### 避免
- ❌ 过于书面化（"综上所述"、"由此可见"）
- ❌ AI腔（"值得注意的是"、"这意味着"）
- ❌ 空洞形容词（"非常"、"特别"、"极其"）
- ❌ 长段落（X平台适合短内容）
