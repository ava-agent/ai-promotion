# 小红书技术分享风格（去标题党版）

## 核心原则

### ❌ 不做
- 情绪化标题（"救命！""挖到宝了！"）
- 博眼球（"月入5万""神仙工具"）
- 夸张数字（"3天学会""效率提升100x"）
- 营销话术（"yyds""绝绝子""快冲"）

### ✅ 要做
- 平实标题（"Trip Agent开发记录""优化过程分享"）
- 技术分享（架构设计、算法选择、踩坑记录）
- 真实数据（性能对比、测试结果）
- 专业视角（架构师身份、工程实践）

---

## 内容定位

### 不是：营销号
❌ "姐妹们快看这个神仙AI工具！"
❌ "用了3个月，真的太香了！"

### 而是：技术博客的口语化版本
✅ "Trip Agent的约束求解器设计"
✅ "从O(n³)到O(n)的优化过程"

---

## 内容类型

### 1. 开发记录类（40%）
**特点**: 记录项目开发过程，有技术深度
**风格**: 技术博客的轻松版本
**受众**: 对技术感兴趣的小红书用户

**示例主题**:
- 「Trip Agent开发周记：第3周」
- 「选型笔记：为什么用CSP而不是贪心算法」
- 「重构记录：把求解时间从500ms降到150ms」
- 「测试发现：20个景点规划的性能数据」

**结构**:
```
标题: [项目名] + [技术点] + [时间/阶段]

开头: 当前阶段的目标
"这周主要在做约束求解器的优化..."

中间: 技术过程
- 遇到的问题
- 尝试的方案
- 最终的选择
- 关键代码片段（简化版）
- 性能数据

结尾: 阶段性成果 + 下一步计划
"目前求解时间稳定在150ms左右，
下周准备做缓存层的优化..."

标签: #开发记录 #算法优化 #Python
```

---

### 2. 技术思考类（30%）
**特点**: 分享技术选型的思考过程
**风格**: 架构师视角，有理有据
**受众**: 有一定技术基础的读者

**示例主题**:
- 「架构决策：为什么不用现成的规划API」
- 「算法选择笔记：TSP问题的几种解法对比」
- 「工程实践：数据验证层应该放在哪里」
- 「性能优化：缓存策略的选择」

**结构**:
```
标题: [技术点] + [思考/决策]

开头: 背景和问题
"在做Trip Agent的时候，
需要解决一个经典问题：
20个景点的最优路径规划..."

中间: 方案对比
- 方案A：贪心算法
  - 优点：快
  - 缺点：质量差
- 方案B：CSP
  - 优点：最优解
  - 缺点：较慢
- 方案C：启发式
  - ...

分析: 为什么选B
"从业务角度考虑，
用户更需要'准'而不是'快'..."

结尾: 实际效果
"生产环境跑了3个月，
平均求解时间127ms，
用户满意度4.6/5.0"

标签: #架构设计 #算法 #工程实践
```

---

### 3. 踩坑记录类（20%）
**特点**: 真实的技术问题排查过程
**风格**: 技术博客风格，有细节
**受众**: 正在学习或遇到类似问题的读者

**示例主题**:
- 「调试笔记：求解器偶尔超时的问题」
- 「踩坑记录：Google Maps API的配额限制」
- 「问题排查：缓存命中率低的原因」
- 「测试发现：节假日营业时间的坑」

**结构**:
```
标题: [问题类型] + [具体问题]

开头: 问题现象
"最近发现求解器偶尔会超时，
正常150ms，有时候会到2s+..."

排查过程:
1. 先怀疑是数据问题
   - 检查了输入数据，正常
   
2. 然后看是否是边界情况
   - 发现景点数量>15时概率增加
   
3. 最后定位到求解器参数
   - 默认最大迭代次数不够
   
解决: 增加迭代次数上限
"把max_iterations从1000调到5000，
超时问题解决，
平均时间从150ms增加到180ms，
可接受"

反思: 经验教训
"生产环境的参数调优很重要，
默认值不一定适合所有场景..."

标签: #调试 #踩坑 #工程实践
```

---

**特点**: 真实使用过的工具和资源

### ✅ 改进后（技术分享）
```
标题：Trip Agent开发记录：约束求解器优化

这周主要在做性能优化。

背景：
Trip Agent的核心是行程规划算法，
20个景点的排列组合有2.4×10¹⁸种可能，
需要找到满足约束的最优解。

问题：
初期的贪心算法虽然快（5ms），
但结果质量一般，用户反馈不够精准。

方案：
改用CSP（Constraint Satisfaction Problem），
用OR-Tools的CP-SAT求解器。

关键代码：
```python
# 定义约束
for i, venue in enumerate(venues):
    model.Add(visit_time[i] >= venue.open_time)
    model.Add(visit_time[i] <= venue.close_time)

# 最大化用户偏好
model.Maximize(
    sum(preference[i] * is_selected[i])
)
```

性能对比：

| 算法 | 时间 | 质量评分 |
|------|------|----------|
| 贪心 | 5ms | 65/100 |
| CSP | 150ms | 92/100 |

选择CSP的原因：
虽然慢了30倍，但质量提升40%，
用户更需要'准'而不是'快'。

优化：
加了缓存层，命中率87%，
平均响应时间降到127ms。

数据：
跑了3个月，求解10000+次，
用户满意度4.6/5.0。

下一步：
继续优化缓存策略，
目标降到100ms以内。

代码开源在GitHub，
有兴趣可以交流。

#开发记录 #算法优化 #Python
```
