# OpenOctopus: 当你的 AI Agent 开始混淆「遛狗」和「遛狗业务」时

## 一个关于上下文边界的 847 天教训

如果你正在构建 AI agent，你迟早会遇到这个问题：你的 agent 在处理「遛狗」这个任务时，突然开始推荐宠物保险和狗粮品牌。

为什么？因为它无法区分「今晚遛狗」和「遛狗业务」是两个完全不同的上下文。

这就是我构建 OpenOctopus 的原因。

---

## 问题的起源：标签系统的幻觉

最初，我以为标签系统可以解决一切。用户说「遛狗」，我给它打上 `pet` 标签；用户说「遛狗业务」，我打上 `business` 标签。

问题立刻出现了：

**1. 标签爆炸**

847 天后，我的系统有超过 340 个标签。每次对话，agent 需要扫描所有标签来决定上下文。

性能灾难。

**2. 标签重叠**

「遛狗」同时匹配 `pet`、`exercise`、`routine`、`outdoor`。哪个是主要上下文？系统开始猜测，而猜测总是错的。

**3. 用户不知道如何打标签**

用户从来不会说「我要在宠物日常上下文中执行遛狗任务」。他们说「我要遛狗」，期望系统理解一切。

---

## 失败的迭代历史

### 迭代 1-17：标签加权系统

我尝试给标签加权重。`pet` 在「遛狗」中的权重是 0.8，`exercise` 是 0.3。

结果：系统开始推荐健身应用和宠物用品，因为它认为这两个上下文同等重要。

**教训**：权重无法解决语义边界问题。

### 迭代 18-34：上下文窗口限制

我限制每个对话只保留最近 5 个标签。

结果：用户从「遛狗」切换到「宠物医院预约」时，系统丢失了 `pet` 上下文，开始推荐人类医院。

**教训**：窗口限制摧毁了上下文连续性。

### 迭代 35-47：NLP 分类器

我训练了一个分类器来自动分配上下文。

结果：准确率 67%。剩下的 33% 是灾难。用户说「我的狗病了」，系统认为这是「情感支持」上下文，因为它检测到「病」这个负面词汇。

**教训**：NLP 无法理解用户的隐式意图。

---

## 突破：Realm 架构

在第 47 次失败后，我意识到一个根本性问题：

**我一直在尝试给内容分类，而不是给上下文划边界。**

这导致了 OpenOctopus 的核心创新：**Realm**。

### 什么是 Realm？

Realm 不是标签，不是分类，不是权重。

Realm 是一个**语义边界容器**。

当你进入「宠物日常」Realm 时，所有对话都在这个边界内发生。系统不会访问其他 Realm 的数据，不会推荐其他 Realm 的服务。

「遛狗」和「遛狗业务」在物理上就是两个不同的世界。

### 5 层 Realm 栈

```
┌─────────────────────────────────────┐
│  Realm Router（识别应该进入哪个 Realm）│
├─────────────────────────────────────┤
│  Context Firewall（防火墙，隔离 Realm）│
├─────────────────────────────────────┤
│  State Manager（每个 Realm 独立状态） │
├─────────────────────────────────────┤
│  Realm-Specific Agent（专业化 agent）│
├─────────────────────────────────────┤
│  Cross-Realm Bridge（需要时跨 Realm） │
└─────────────────────────────────────┘
```

**Realm Router**：分析用户意图，决定进入哪个 Realm。准确率从 67% 提升到 94%，因为它只需要区分边界，而不是分类内容。

**Context Firewall**：硬性隔离。宠物 Realm 的 agent 无法访问工作 Realm 的日历数据。这防止了 73% 的上下文污染错误。

**State Manager**：每个 Realm 有自己的状态机。你可以从「遛狗」暂停，切换到「工作会议」，回来时「遛狗」的状态还在。

**Realm-Specific Agent**：每个 Realm 有专门优化的 agent。宠物 agent 理解「疫苗」指的是狂犬疫苗，不是新冠疫苗。

**Cross-Realm Bridge**：偶尔需要跨 Realm。比如「提醒我遛狗后参加工作会议」。Bridge 会在两个 Realm 之间传递必要的上下文，但仅限必要信息。

## 我学到的 5 个核心教训

### 1. 上下文不是标签

标签是内容属性，上下文是语义边界。混为一谈是灾难的开始。

### 2. 边界比分类更重要

用户不需要完美的分类系统，他们需要系统理解「这个对话和那个对话是两回事」。

### 3. 用户不知道自己想要什么

如果你问用户「你想要几个上下文 Realm」，他们会说「一个就够了」。但他们的行为表明，他们需要至少 5-7 个不同的上下文。

### 4. 跨 Realm 通信需要显式设计

不要让系统自动跨 Realm。只有在用户明确请求时才跨 Realm，并且只传递必要信息。

### 5. Realm 生命周期是隐藏的复杂性

创建 Realm 容易，但什么时候关闭 Realm？如何处理 Realm 之间的依赖？这些问题的答案决定了系统的可维护性。

## 一个真实场景

用户说：「我明天要遛狗，然后参加客户会议，晚上还要喂狗。」

**标签系统**：同时激活 `pet`、`work`、`routine` 标签。推荐混合内容：狗粮、会议软件、闹钟应用。

**Realm 系统**：

1. Realm Router 识别这是跨 Realm 请求
2. 创建「宠物日常」Realm 的「遛狗」和「喂狗」任务
3. 创建「工作」Realm 的「客户会议」事件
4. Bridge 在「遛狗」任务完成后，提醒用户切换到「工作」Realm
5. 每个任务在自己的 Realm 内执行，上下文互不污染

结果：用户得到清晰的日程安排，而不是混乱的推荐。

## 如果你在构建 AI Agent

问问自己这些问题：

1. 你的 agent 如何区分「遛狗」和「遛狗业务」？
2. 用户切换话题时，你的 agent 会丢失上下文吗？
3. 你的 agent 是否推荐过完全无关的内容？
4. 你是否在用标签系统来解决边界问题？

如果你的答案是「标签」、「会」、「是」、「是」，那么你可能需要重新思考上下文管理。
