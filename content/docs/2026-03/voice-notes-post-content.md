从0到1构建 Voice Notes Assistant：语音AI的技术边界与工程实践

---

## 1. 项目概述：我们试图解决什么问题？

在日常工作和学习中，我们经常会遇到这样的场景：
- 开会时突然有灵感，但打字太慢
- 开车时想到重要事项，无法记录
- 访谈、讲座内容太多，事后整理耗时

**Voice Notes Assistant** 的诞生源于一个简单的信念：**语音应该成为人与AI之间最自然的交互方式**。我们的目标是打造一个能够实时转录、智能摘要、快速检索的语音笔记系统。

---

## 2. 核心挑战与架构设计

### 2.1 实时性 vs 准确性的平衡

语音转录面临的首要挑战是延迟。用户说话的同时就想看到文字，但高质量的转录往往需要更多计算时间。

**我们的解决方案：分层转录架构**

```
用户语音 → 流式VAD检测 → 快速预览模型(Whisper-tiny) → 实时显示
                        ↓
                段落结束 → 精确转录模型(Whisper-medium) → 修正更新
                        ↓
                长时间录音 → 云端大模型校对 → 最终文本
```

这种"渐进式精化"的策略让用户先看到草稿，再获得精校版本，体验上几乎无感知。

### 2.2 说话人分离（Diarization）的工程化

多人会议场景下，区分谁在说话是个难题。我们尝试了三种方案：

| 方案 | 延迟 | 准确率 | 资源消耗 | 适用场景 |
|------|------|--------|----------|----------|
| 云端API | 2-3s | 95%+ | 高 | 正式会议 |
| 本地轻量模型 | 500ms | 85% | 低 | 日常记录 |
| 声纹预注册 | 实时 | 90% | 中 | 固定小组 |

最终我们采用**自适应策略**：根据设备性能和网络状况自动选择模型。

### 2.3 智能摘要的Prompt工程

转录只是第一步，如何让AI真正"理解"内容并生成有价值的摘要？

我们经过上百次迭代，总结出有效摘要的关键要素：

**Prompt设计原则：**
1. **结构化输出**：要求模型按"要点-行动项-问题"格式输出
2. **角色设定**："你是一位专业的会议记录员..."
3. **示例引导**：提供2-3个高质量示例
4. **长度控制**：明确要求字数范围

**一个有趣的发现：** 在Prompt中加入"忽略语气词和重复内容"，能让摘要质量提升30%以上。

## 3. 技术选型的得与失

### 3.1 转录引擎的选择

我们对比了当前主流方案：

- **Whisper (OpenAI)**：准确率高，支持多语言，但本地部署资源消耗大
- **FunASR (阿里)**：中文优化好，轻量版本可在移动端运行
- **Paraformer (达摩院)**：流式识别延迟低，适合实时场景

**最终方案：** Whisper-medium作为主力，FunASR作为移动端降级方案。

### 3.2 向量检索的踩坑实录

为了让用户能"用自然语言搜索语音内容"，我们实现了语义检索功能：

```
语音 → 转录文本 → Embedding模型 → 向量数据库 → 语义搜索
```

**踩过的坑：**
1. ** embedding模型选择**：通用模型（如text-embedding-3-small）对口语化内容效果不好，最终选用针对对话优化的模型
2. **分片策略**：长录音直接整体嵌入会导致检索粒度太粗，我们采用滑动窗口分片 + 层次化索引
3. **增量更新**：录音进行中如何实时更新索引？我们采用"片段临时索引 + 结束合并"的策略

### 3.3 本地优先 vs 云端协同

隐私是语音产品的敏感点。我们坚持**本地优先**原则：

- 录音文件本地加密存储
- 基础转录支持完全离线
- 可选开启云端增强功能
- 所有数据上传前明确授权

这种设计增加了开发复杂度，但赢得了用户信任。

## 4. 产品化过程中的关键决策

### 4.1 极简交互设计

语音产品的交互必须极简。我们的设计哲学：

**"一键开始，自动结束"**

- 点击即开始录音
- 智能检测停顿自动分段
- 无需手动停止，系统判断说话结束

**"所见即所得"**

- 转录文本实时可见
- 关键信息自动高亮
- 支持随时插入标记

### 4.2 多场景适配

不同场景对语音笔记的需求差异巨大：

| 场景 | 核心需求 | 技术重点 |
|------|----------|----------|
| 会议记录 | 多人区分、行动项提取 | 说话人分离、结构化输出 |
| 访谈整理 | 长文本处理、关键信息标注 | 长上下文理解、实体识别 |
| 灵感速记 | 快速启动、极简操作 | 低延迟唤醒、本地优先 |
| 学习笔记 | 知识点关联、复习提醒 | 知识图谱、间隔重复 |

我们采用**场景模式**设计，用户选择场景后，系统自动调整转录参数、摘要Prompt和存储策略。

### 4.3 错误处理的用户体验

语音识别不可能100%准确。如何让错误不破坏用户体验？

**我们的做法：**
1. **置信度可视化**：低置信度的词用虚线标注
2. **一键重听**：点击任意文字可播放对应音频片段
3. **快速修正**：长按单词可直接编辑，系统自动学习修正模式
4. **上下文补全**：AI根据上下文自动建议修正

## 5. 性能优化实战

### 5.1 移动端功耗控制

持续录音 + AI处理 = 手机电量杀手

**优化策略：**
- **动态采样率**：静音时降至8kHz，说话时提升至16kHz
- **硬件加速**：利用NPU进行VAD检测和Embedding计算
- **后台策略**：锁屏后降低转录频率，优先保证续航

### 5.2 冷启动优化

模型加载是启动延迟的主要来源：

```
优化前：启动 → 加载Whisper(2.3s) → 加载Embedding(1.1s) → 就绪
优化后：启动 → 加载Tiny模型(0.3s) → 就绪 → 后台加载完整模型
```

用户感知的启动时间从3.4s降至0.3s。

### 5.3 存储空间管理

原始录音 + 转录文本 + 向量索引 = 存储占用快速增长

**生命周期策略：**
- 7天内：完整数据本地保留
- 30天内：保留转录文本，原始录音云端可选
- 90天后：仅保留摘要和关键片段
- 用户可标记"永久保存"突破限制

## 6. 学到的经验与反思

### 6.1 技术债的代价

早期为了快速验证，我们直接在主线程进行转录。随着模型升级，卡顿问题愈发严重。重构为异步架构花了整整两周。

**教训：** 即使是MVP，也要考虑核心模块的可扩展性。

### 6.2 用户反馈的意外价值

我们曾认为"实时转录"是最重要的功能。但用户反馈显示：
- 60%的用户更看重转录后的整理功能
- 25%的用户希望支持导入已有录音
- 只有15%的用户真正需要实时性

这让我们调整了功能优先级，资源投入产出比提升了2倍。

### 6.3 边缘案例的处理

语音产品面临的边缘案例超乎想象：
- 方言识别失败率比普通话高40%
- 背景噪音>60dB时准确率急剧下降
- 快速说话(>180字/分钟)漏字率增加

**应对：** 建立用户反馈闭环，持续收集bad case优化模型。
