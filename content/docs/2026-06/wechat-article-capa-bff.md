# 用了Capa-BFF三个月，聊聊零成本BFF到底香不香

![](https://images.unsplash.com/photo-1551288049-bebda4e38f71?auto=format&fit=crop&q=80&w=1200)

说实话，我当时看到这个项目的时候眼睛都亮了。零成本BFF？这不是解决我们团队痛点的银弹吗？

我们团队一直做To B业务，前端好几个端——Web、小程序、APP，每个端要的数据格式都不一样。后端每次都要改接口，改到后来大家都烦了。服务端说前端事多，前端说后端不懂适配，中间夹着BFF层没人愿意维护，开发效率慢得要死。

所以当我看到Capa-BFF宣传"零成本BFF"的时候，我当场就决定拉到生产环境试试水。反正不要钱，大不了踩坑了再换回去呗。三个月用下来，心态从激动到平静再到务实，今天跟大家好好唠唠。

## 背景：为什么我们需要BFF？

先说说我们为什么要上BFF吧。可能有些同学还不太清楚BFF是啥，简单说就是前端后端之间的那层聚合适配。

传统开发模式下，后端写好接口，前端直接调。但多端场景就麻烦了——Web要的字段多，移动端要精简，小程序还有各种大小限制。后端不想重复开发，前端自己聚合又太重，怎么办？加一层BFF呗。

问题来了，加一层BFF成本真不低。你要写代码、要维护、要部署，出问题了你还得调试。GraphQL方案学习曲线又陡，团队没人会，招人还贵。

所以市面上一直喊着要低成本BFF，真正能落地的没几个。Capa-BFF这个项目是携程黑客松金奖作品，思路挺新奇的——靠DSL动态调用，不用写代码就能搞定聚合裁剪。我当时就想，这思路要是成了，那真能省不少事。

## 核心思路：零成本到底是怎么做到的

Capa-BFF的核心其实就是三个设计原则，理解了这三点你就明白它为啥敢说零成本了。

第一个是动态化调用。不管你后端有多少服务，不管接口怎么变，Capa-BFF都不需要改代码。它靠一个三元组`(appId, methodName, data)`来定位服务，前端想要什么直接在DSL里写清楚就行。

第二个是数据别名映射。这个设计我觉得挺聪明的——把不同服务的响应映射到统一的别名空间，你要什么字段就留什么，不需要的直接裁掉。相当于给数据做了个SPA瘦身。

第三个是基于ID的增强功能。对服务调用或者字段可以加函数增强，想做个格式转换啊、计算啊什么的，直接在DSL里定义就行，不用改代码。

整个核心流程也很清晰：

```
DSL 解析 → 依赖分析 → 拓扑排序 → 并行/串行调用 → 响应裁剪 → 结果返回
```

给大家看一段真实的DSL配置，感受一下：

```json
{
  "user-service": [
    "getUserInfo": (
      request: {
        "userId": 123
      },
      response: {
        "result.user.id": "user.id",
        "result.user.name": "user.name",
        "result.user.email": "user.email"
      }
    ),
    "getUserOrders": (
      request: {
        "userId": "${user.id}"
      },
      response: {
        "result.orders[*].id": "orders[*].id",
        "result.orders[*].amount": "orders[*].amount",
        "result.orders[*].createTime": "orders[*].createTime"
      }
    )
  ],
  "order-service": [
    "getOrderDetail": (
      request: {
        "orderId": "${orders[0].id}"
      },
      response: {
        "*": "latestOrder"
      }
    )
  ]
}
```

看到没？就写这么一段JSON配置，BFF层就搞定了。真不用写Java代码。依赖关系它自己会分析，有循环依赖还能帮你检测出来，调用顺序自动做拓扑排序，能并行的它绝不串行。

这个思路说实话挺颠覆的。传统BFF你得写Controller、写Service、拼结果，这里配置完直接用，开发效率提升不是一星半点。

## 实战：我是怎么集成到Spring Boot的

说了这么多，来看看实际怎么用吧。我们项目是Spring Boot + Dubbo，集成起来其实不复杂。

首先引入依赖：

```xml
<dependency>
  <groupId>com.capa-cloud</groupId>
  <artifactId>capa-bff-spring-boot-starter</artifactId>
  <version>1.0.0</version>
</dependency>
```

然后写个配置类：

```java
@Configuration
@ComponentScan("com.capa.bff")
@EnableCapaBFF
public class CapaBFFConfig {
    
    @Bean
    public DSLParser dslParser() {
        return new DefaultDSLParser();
    }
    
    @Bean
    public InvocationExecutor invocationExecutor() {
        return new DubboInvocationExecutor();
    }
}
```

最后把DSL配置文件放到`classpath:bff/`目录下，启动项目就完事儿了。访问接口就是：

```
POST /bff/{config-name}
```

就这么简单。我当时从引入到跑通第一个案例，前后不到一小时。说真的，那一刻我觉得这玩意儿真神了。

我们第一个场景是用户中心聚合。原来用户信息分散在三个服务里，前端要调三次接口，现在前端只需要调一次Capa-BFF，它帮你聚合好再返回。页面加载速度直接提升了两倍，原来要等三个请求回来才能渲染，现在一次搞定。

那天晚上我回去都有点激动，感觉找到宝藏了。结果第二天踩坑就开始了。

## 踩坑：那些让我心态崩了的时刻

说实话，踩坑的时候我真想把这玩意儿删掉回滚。不是说它不好，是你得知道它的坑在哪里，不然上线了真要命。

第一个坑就是**依赖调试真难**。

Capa-BFF帮你自动分析依赖、排序、调用，听起来很美好，但调用链出问题了你怎么调试？它把很多步骤黑盒掉了，哪一步失败了，打日志都不好打。有次我这边一直返回空结果，查了两个小时才发现是DSL里别名写错了，路径多了个点。这种错误编译器不帮你检查，全靠你自己眼睛看。

我当时心态都崩了，与其在这里debug，我还不如自己写两行代码呢。

第二个坑是**动态调用性能损耗**。

我做了个压测，直接调用Dubbo接口平均耗时是20ms左右，经过Capa-BFF一层动态调用，耗时涨到了50-80ms。不算特别夸张，但对于敏感接口这个延迟还是能感觉到的。

尤其是嵌套依赖多的时候，解析+反射+多次调用，耗时还会往上走。我们那个订单聚合接口，依赖了三个服务，平均耗时快150ms了。比原来直接拼起来慢了不少。

第三个坑是**DSL写多了真的乱**。

一开始就一两个配置，看着挺清爽。后来业务发展，几十个BFF接口，每个接口几十行JSON配置，找问题的时候翻配置翻得头晕。没有语法提示，没有类型检查，复制粘贴容易错，改的时候牵一发而动全身。

有次产品改了个字段名，我忘了在DSL里改，结果线上出问题。排查了半小时才发现就是个字符串写错了。这种低级错误，在Java里编译就过不了，在这里直接跑到线上才爆。

第四个坑是**监控和日志基础太薄弱**。

开源版本基本没什么监控，哪个服务调用慢了？失败率多少？哪个DSL配置热点最高？啥都没有。出了问题全靠你自己打日志分析。我们线上出了一次性能问题，我加了半天才定位到是某个DSL里循环依赖导致重复调用。

要是生产环境用，监控这一块你得自己补不少代码。

第五个坑是**学习曲线其实没那么平**。

项目宣传说零成本上手，说实话那是指跑通Demo。真要用好，你得理解它的依赖分析原理，得理解别名映射规则，得知道怎么写嵌套依赖，这些都得花时间踩坑。我团队那个新来的前端同学，看了文档半天还是写不对DSL。

所以啊，零成本说的是开发成本零，不是学习成本零。这一点千万别被忽悠了。

## 数据：压测结果说话，到底性能怎么样

我专门做了个简单压测，给大家看看真实数据。机器是普通的4C8G云服务器，JDK11，Spring Boot 2.7。

| 场景 | QPS | 平均耗时 | P99耗时 |
|------|-----|----------|---------|
| 直接调用Dubbo（单服务） | 8500 | 18ms | 45ms |
| Capa-BFF（单服务无依赖） | 5200 | 48ms | 110ms |
| Capa-BFF（三服务并行依赖） | 3800 | 85ms | 180ms |
| Capa-BFF（三服务串行依赖） | 2600 | 150ms | 280ms |

数据放在这里，大家自己判断。说实话，对于中小项目来说这个性能完全够用。我们页面响应要求不高，100-200ms用户感知不明显。但如果你做的是高并发首页，那这个损耗可能就有点吃不消了。

内存占用方面，加载50个DSL配置，占用内存不大，也就几十兆。启动速度也没明显变慢，这点做得还可以。

我也整理一下优缺点，给大家参考：

**优点：**
- 开发真的快，配置完就能用，不用写Java代码
- 依赖自动管理，拓扑排序帮你做了，能并行绝不串行
- 数据裁剪灵活，想要什么字段就要什么，多端适配真的方便
- 社区还算活跃，黑客松出来的项目，思路很新颖
- 对Spring Boot集成友好，starter开箱即用

**缺点：**
- 调试困难，黑盒太多，出问题不好定位
- 动态调用有性能损耗，高并发场景要谨慎
- DSL没有类型检查，拼写错误线上才发现
- 监控日志薄弱，生产环境需要自己补
- 复杂场景DSL会变得很大，维护成本上升

## 感悟：技术选型别被概念忽悠了

用了三个月，最大的感悟是什么呢？其实就是——没有银弹。

Capa-BFF思路真的很棒，解决了很多传统BFF的痛点，但它不是万能的。它适合你，不一定适合我。

说实话，我一开始真被"零成本BFF"这个概念冲昏头脑了。觉得这玩意儿好，能解决一切问题，恨不得全团队都推广。踩了几个坑之后才冷静下来——零成本是有前提的。

如果你的团队是中小项目，业务迭代快，对性能要求不是特别极致，那Capa-BFF真的能帮你省很多事。原来要写一个BFF接口一天，现在十分钟配置完，香不香？

但如果你是超大型项目，高并发场景，对延迟要求特别敏感，团队又有专门的BFF团队维护，那可能还是老老实实写代码吧。这点性能损耗，加上调试维护成本，真不如自己写来得踏实。

还有一点挺重要的——你的团队能不能接受DSL配置这种开发方式。如果团队都是Java工程师，写惯了强类型代码，突然让大家写JSON配置找错误，很多人会不适应。学习成本其实比你想象的高。

我现在的使用场景是——内部工具、后台管理系统、对延迟不敏感的业务接口，都用Capa-BFF，开发速度提升明显。核心链路、高并发接口，还是自己写BFF，稳一点。

说白了，技术选型就是权衡。没有最好的技术，只有最合适的。你接受它的优点，也要接纳它的缺点。

我现在反而觉得，Capa-BFF最有价值的不是它的代码，是它这个思路——BFF一定要写代码吗？能不能靠配置搞定？这种思考其实比代码本身更重要。

也许以后会有更好的方案出来，但至少现在，Capa-BFF给了我们一种新选择。对于中小团队来说，能快速落地解决问题，就已经够了。
