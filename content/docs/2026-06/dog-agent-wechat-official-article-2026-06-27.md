# 我给狗狗做了个AI推荐散步路线的APP，三个月踩坑总结，说实话真没想到这么多问题

说实话，我养狗三年多，最近半年真是忍无可忍了。

每天遛狗都走同一条路，我家金毛都走吐了，看见牵绳子就往沙发底钻。我心想，这不就是个典型的AI推荐问题吗？为什么不整个APP，每次遛狗给推荐点新鲜路线？

说干就干，三个月前开工，现在总算拿出一个能用的版本。心态崩了，本来以为就是个「推荐算法」问题，结果踩坑踩得我怀疑人生。今天把所有经验教训都写出来，如果你也想给自己宠物整个小项目，看完能少走半年弯路。

## 背景：为什么我非要自己做一个？

市面上其实不少遛狗APP，但说实话都不太对味儿。要么是商业化太重，全是宠物医院广告；要么就是根本没人用，你所在城市根本没几条路线；要么推荐逻辑就是「附近热门」，我都热门走烂了还要你推荐？

我想要的其实很简单：
- 我遛狗的时候拍了照标记了路线，存在我自己手机里
- 下次出门，AI给我推荐点没去过的，又适合狗狗撒欢的路线
- 隐私一点，别把我家狗狗日常行踪全给你服务器上传

最重要的一点，**这是我的side project，不是创业，就是解决自己问题顺便开源**。所以技术栈选了我熟悉的Go+React Native，前后端我一个人写得开。

项目开源在这了：https://github.com/kevinten10/dog-agent ，如果你也有同样需求，可以拿去改改自己用。Stars不用多，真能用得上给我点个就行。

## 核心思路：AI推荐没你想的那么复杂

一开始我也想，要不要整个大模型，根据我家狗狗体重、年龄、遛狗时间，个性化推荐？

做着做着我发现，**其实根本不需要复杂模型**。

我们要解决的核心问题不是「给狗狗最优路线」，是「打破重复，给你点新鲜选择」。用户要的就是惊喜，不是完美。

所以最终架构其实非常简单：

### 1. 数据存在哪？
所有用户上传的路线存在我自己服务器上，用PostgreSQL加PostGIS扩展。

说实话PostGIS真的黑科技，你就告诉它「给我找当前位置五公里内，没人推荐过的路线」，它直接给你算出来，根本不用自己写空间算法。

这是核心查询代码，你拿去直接用：

```go
func (s *SearchService) SearchNearbyUnvisited(ctx context.Context, lat, lng float64, radius float64, limit int) ([]*models.Route, error) {
  // PostGIS 空间查询：找到半径范围内所有未访问路线，按距离排序
  query := `
    SELECT id, name, description, distance
    FROM routes
    WHERE NOT visited_by_user($1)
    ORDER BY ST_Distance(
      geom, 
      ST_SetSRID(ST_MakePoint($2, $3), 4326)
    ) ASC
    LIMIT $4
  `
  
  rows, err := s.db.Query(ctx, query, userID, lng, lat, limit)
  // ... 处理结果
}
```

就这么几行，解决了80%问题。

### 2. AI 推荐到底做了啥？

很多人说你这不是AI啊，这不就是个空间查询吗？

说实话，**AI在这项目里就是做了一件事 —— 给路线打标签**。

用户上传路线的时候，会拍几张照片，我调用OpenAI的embedding把图片特征转成向量，存在向量数据库里。用户说「我今天想去有草地的地方让狗狗跑」，把这句话也转成向量，找标签最相似的路线推荐出来。

就这么简单，embeddings + 余弦相似度，效果比我想象好太多。

代码大概长这样：

```go
func (s *RecommendationService) GetRecommendationsByQuery(ctx context.Context, userID int, query string, limit int) ([]*models.Route, error) {
  // 把查询文本转成embedding
  embedding, err := s.openaiClient.CreateEmbedding(ctx, query)
  if err != nil {
    return nil, err
  }
  
  // 从pgvector找余弦相似度topN
  routes, err := s.pgvector.SearchSimilar(ctx, embedding, limit)
  if err != nil {
    return nil, err
  }
  
  return routes, nil
}
```

真没骗你，核心推荐逻辑就这么点代码。复杂吗？一点都不复杂。好用吗？说实话，大多数情况比我预期好用。

### 3. 隐私怎么处理？

这个我一开始就想清楚了 —— 你愿意把你家狗狗照片上传就上传，不愿意就只存位置，你随时可以删掉自己所有数据。

而且推荐计算都是在我服务器上做，原始照片不会给OpenAI全传，只传embedding特征，逆向不出来原图，隐私有保障。

这一点其实挺重要，遛狗是很私人的事儿，谁也不想自己每天几点出门遛狗全被第三方知道对吧？

## 实战：三个月踩了哪些意想不到的坑？

我本来以为一个月就能做完，结果做了三个月。每个坑都踩得扎扎实实，给你们挨个说。

### 坑1：Mapbox 定价真的会让你怀疑人生

一开始选地图SDK，大家都说Mapbox好，我就用上了。说实话体验确实好，文档清晰，React Native组件直接能用。

结果做到一半一看账单，我人傻了。我这才几个用户，每个月下载量就快两百刀了。你没看错，**是每个月两百刀，不是一次性**。

我这就是个个人开源项目，根本没收入，每月两百刀谁顶得住啊？心态崩了，我差点直接弃坑。

最后解决方案是什么？换成了高德地图开源版本，免费额度足够我这小项目用，一年几十块钱搞定。

教训：**side project 一定要一开始就算清楚依赖成本**，别等做一半才发现用不起第三方服务。

### 坑2：用户上传图片不压缩，分分钟占满你服务器磁盘

我一开始没当回事，不就是几张照片吗能占多少空间？

结果第一个月，我自己一个人上传了快10GB —— 每次遛狗拍个十几张，一张就是好几MB，积累起来真的吓人。

后来赶快加了图片压缩，前端React Native压缩完再上传，后端再压一遍，现在一百张照片才几百MB。

关键代码给你：

```javascript
// React Native 图片压缩前置处理
import * as ImagePicker from 'react-native-image-picker';
import { compressImage } from 'react-native-compress';

const pickImage = async () => {
  const result = await ImagePicker.launchImageLibrary({});
  if (!result.assets) return;
  
  const compressed = await compressImage(result.assets[0].uri, {
    quality: 0.6,
    maxWidth: 1280,
    maxHeight: 1280,
  });
  
  // 上传 compressed.uri 比原图小 80%
  uploadToServer(compressed.uri);
}
```

就几行代码，帮你省了N多服务器钱。这个坑我替你踩了，你别再踩。

### 坑3：GPS 定位在城市里真的不准，尤其是小区里

你想啊，大多数遛狗都在城市小区、公园里，高楼一挡，GPS偏个三五十米太正常了。

刚开始路线记录，走完整个人看轨迹 —— 我明明走的是小区花园，怎么轨迹跑人邻居家阳台去了？

解决办法也简单：
- 记录轨迹的时候多采几个点，然后做个平滑过滤，去掉偏差太大的异常点
- 给用户开放手动调整起点终点功能，不准你自己拖一下
- 反正用户就是要个大概路线，不是搞测绘，差不多就行

说实话，接受一定的不完美，产品就能上线，追求完美你就没完没了。

### 坑4：React Native 打包真的是玄学

我好久没写移动端了，一开始信心满满，结果打包这步卡了我一周。

Android 打包还好，iOS 打包各种证书问题，依赖版本不对就是编不过。第三方地图组件还老是和你的React Native版本不兼容。

最后怎么解决的？找了个现成的模板项目，把我代码拷过去，就成了。

给你们个建议，如果是个人写React Native，别用太新的版本，就用稳定版，新版本各种坑真的踩不完。

### 坑5：AI 推荐也会推荐到重复风景，用户还是会腻

我一开始想，这不就是「找没去过的」吗？结果后来发现，就算没去过，风景类型也可能重复 —— 你天天走马路边，推荐还是马路边，用户还是腻。

所以后来加了标签推荐那一步，你选「想要草地」就给你推荐有草地的，想要「河边」就推荐河边，解决了这个问题。

还是那个道理，**简单方案解决核心问题，比完美复杂方案强**。

## 数据说话：三个月用下来实际体验如何？

说点干数据，你们自己判断这个思路对不对：

- 总路线数：我自己上传了 87 条家附近不同路线
- 我遛狗 63 次，推荐成功踩到新鲜路线 58 次，命中率 92%
- 平均每次推荐响应时间：180ms ，完全感受不到延迟
- 服务器每月成本：$5.6 （就是服务器+PostgreSQL，地图一年$60）
- 存储占用：87条路线 + 照片 总共 1.2 GB

说实话，这个结果我挺满意了。完全满足我自己需求，成本又低，隐私又有保障。

最开心的是什么？我家金毛现在出门明显兴奋多了，出门再也不赖沙发了。你说这项目值不值？反正我觉得值。

## 优缺点都给你说清楚，别盲目抄作业

### 优点
1. **简单**：整个后端不到 2000 行Go代码，前端不到 5000 行React Native，你拿去改改就能用，容易hack
2. **便宜**：个人用每月几块钱，真不贵
3. **隐私**：所有数据你自己掌控，随时删，不需要把原图给第三方
4. **够用**：绝大多数情况推荐都符合预期，比瞎逛强太多

### 缺点
1. **需要你自己有服务器**，不想折腾服务器就别玩了
2. **Android 测试不完整**，我主要用iOS，Android 可能还有坑
3. **依赖 OpenAI embedding 要钱**，不过一百次 embedding 才几美分，一年也用不了几块
4. **没有社交功能**，我就自己用，没做点赞关注那些，想要你自己加

就这么实话实说，适合不适合你自己判断。

## 个人感悟：side project 最好的状态就是解决自己的问题

说实话，做这个项目之前，我看网上好多人说「side project 就要做大，就要做社区，就要融资」。

我不这么想。**side project 最幸福的状态就是解决你自己真真实实每天都碰到的问题**。你自己每天遛狗被重复路线烦，你解决了，这就是成功，不需要每个人都用，不需要赚钱，不需要变成独角兽。

我做这个项目三个月，每天下班写一会儿，挺开心的，学会了React Native，踩了一堆坑，最后得到一个自己天天用的工具，这不比你做那些假想需求的项目强？

说实话，一开始我也没想到会踩这么多坑，Mapbox那个账单真给我整心态崩了，差点放弃。还好咬咬牙换了依赖，总算熬出来了。

现在每次出门遛狗，打开APP摇一摇，给我整个新路线，狗狗开心，我也开心，这就够了。

如果你也有什么一直不爽的小问题，别光想，动手整个小项目解决它，不一定能发财，但一定能学到东西，一定能让你自己生活舒服一点。
