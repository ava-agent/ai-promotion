
# 用了Capa-Java三年，说说混合云开发的真实感受

说实话，我一开始接触混合云开发的时候，心态其实有点崩。

我之前在公司做项目的时候，就遇到过这么个事儿：业务需要同时跑在阿里云和AWS上，两边的服务发现、配置管理、消息队列API全不一样。一开始想着那就分别写两套代码算了，结果写着写着发现维护成本高得离谱——改一个逻辑需要两边都改，测试两遍，上线还得错开时间，我那段时间加班加得都快不认识女朋友了😂

后来听朋友说Capa-Java这个项目，抱着试一试的心态用了，这一用就是三年。今天就跟大家聊聊真实感受，好的坏的都说清楚，给有需要的朋友做个参考。

## 先说说这到底是个啥

可能有些朋友还没听说过，简单说一下：Capa-Java是一个面向混合云/多云场景的多运行时SDK，说白了就是帮你写一次代码，就能跑在不同云厂商的环境上。

这个思路其实跟Dapr有点像，但区别在于Dapr是Sidecar模式，而Capa走的是Rich SDK路线。我觉得这个选择挺务实的——说实话，很多企业现有的Java系统想一步切到Sidecar架构真的太难了，改动太大，风险也高。Rich SDK这种方式就能平滑过渡，对老系统友好很多。

它的核心设计其实很简单：**标准API + 可插拔的SPI实现**。你写业务代码的时候只需要依赖Capa的标准API，不用管底层具体是阿里云还是AWS还是K8s，部署的时候引入对应的SPI实现包就行。这样真的省了太多事。

给大家看一段真实可运行的代码，感受一下实际写起来是什么样：

```java
import group.rxcloud.capa.sdk.rpc.CapaRpcClient;
import reactor.core.publisher.Mono;

// 只依赖Capa标准API，不依赖具体云厂商的SDK
public class BusinessService {
    
    private final CapaRpcClient rpcClient;
    
    // 构造注入，具体实现由SPI加载
    public BusinessService(CapaRpcClient rpcClient) {
        this.rpcClient = rpcClient;
    }
    
    // 调用另一个服务，不管底层在哪个云
    public Mono<String> callAnotherService(String message) {
        // SERVICE_APP_ID 是目标应用ID，"say" 是方法名
        return rpcClient.invokeMethod(
            "order-service", 
            "say", 
            message, 
            null, 
            null, 
            String.class
        );
    }
    
    // 同步调用也支持，block一下就行
    public String callSync(String message) {
        return callAnotherService(message).block();
    }
}
```

代码是不是看着挺干净？你不用管底层是用Dubbo还是Spring Cloud还是阿里云的服务发现，Capa都给你封装好了。改个依赖包就能切换环境，对开发者来说真的友好。

## 实际用下来，哪些地方真的好用

我用了三年，在两个生产项目落地过，说几个我觉得确实解决痛点的地方。

第一个就是**迁移成本真的低**。我第一个项目是把一个老的Spring Cloud应用从私有云迁移到公有云，按照以前的玩法，至少得改两三百行代码，服务发现、配置都得动。用Capa的话，其实就是改了几个依赖，业务逻辑几乎没动，前后花了不到两天就上线了。

我之前也踩过同样的坑——为了多云改造，硬着头皮改了半个月代码，结果上线后各种不兼容，熬夜改了一周才稳定。现在有了Capa这种方案，真的能少踩很多坑。

第二个好处是**团队学习成本低**。其实我们团队很多人对Dapr那种Sidecar架构不是特别熟，要搞的话还得重新学运维、监控、网络这些东西。但Capa就是个普通的Java SDK，扔到Maven里就能用，开发同学几乎不用额外学习就能上手。这点对于中小团队来说太重要了——毕竟不是每个团队都有专职的云原生架构师。

第三个我觉得挺不错的就是**API设计跟社区走**。Capa的API定义是放到独立仓库的，跟Dapr/Layotto这些项目保持同步，不会搞一堆自己的私货。这样以后想切到Dapr也容易，不会被绑定死。我觉得这点挺良心的，毕竟谁也不想被某个项目绑架。

## 实话实说，哪些地方其实不太舒服

说了这么多好话，也得说说缺点，实事求是嘛。毕竟没有完美的东西，Capa也一样。

第一个问题就是**目前生态还不够完善**。像数据库、定时任务这些特性还在Alpha阶段，生产环境用不了。我实际项目中，SQL这块还是得自己写，没能用上Capa的抽象。如果你需要用到比较全的功能，可能现在还满足不了，这点得有心理准备。

第二个是**文档其实有点欠缺**。说实话，就是能跑通示例，但遇到一些边缘问题，查文档找不到答案，得自己去翻源码理解。我刚用的时候，就卡在SPI加载那一步，文档没说清楚依赖怎么排，折腾了大半天最后还是看源码才搞定。要是文档能更详细点就好了。

第三个就是**性能对比原生调用确实有损失**。我之前压测过，大概比直接调用原生API慢个5%-10%左右，绝大部分场景其实没啥影响，但如果你对延迟特别敏感，那可能就得好好测试一下了。不过话又说回来，本来抽象就会有性能损耗，这事儿也没办法，就看你能不能接受了。我觉得大部分业务场景这点损失完全可以接受，换来了开发效率提升还是划算的。

## 和Dapr比，到底该选哪个

很多朋友肯定会问，既然思路差不多，那选Capa还是Dapr？我用了这么久，也有点自己的看法，不一定对，给大家参考。

**选Capa-Java比较适合这些情况：**
- 你有一堆老的Java系统需要渐进式迁移到混合云，不想一次改太多
- 团队规模不大，没人专门搞Sidecar运维那一套
- 你想要低侵入性，改动小，快速上线
- 你对性能要求不是极端苛刻，能接受一点额外损耗

**选Dapr更适合这些情况：**
- 你从头开始做新项目，能接受全新架构
- 多语言栈，不只是Java
- 团队有运维能力，能hold住Sidecar
- 你想要完整的生态和社区支持

我觉得吧，没有谁好谁坏，就是适用场景不一样。Dapr是未来的方向，但Capa解决了现在很多老系统迁移的实际问题，存在就是合理的。

## 三年用下来，最大的感受是什么

其实我最大的感受就是，混合云这个东西，真的不是每家公司都需要。我见过很多团队，为了"多云"这个架构漂亮，硬生生把简单问题复杂化，最后投入产出比极低。

我觉得吧，如果你的应用真的需要跑在多个云上，或者未来有这个可能，那用Capa这种方案平滑过渡真的挺好。但如果你的应用这辈子估计都只会跑在一个云上，那真没必要瞎折腾——简单才是最美的，直接用云厂商原生API就完事了。

可能因人而异吧，我现在做技术选型越来越务实了：能解决问题，成本低，就是好方案，不一定非得追最时髦的架构。

## 总结一下

说了这么多，总结一下：Capa-Java是一个挺务实的混合云Java SDK方案，对于需要渐进式迁移的老Java系统来说真的挺好用，解决了实际问题，迁移成本低，学习成本也不高。但它也有缺点，生态还不完善，文档需要加强，性能也有一点损耗。

如果你现在正卡在老系统混合云迁移这块，不妨试试，反正就是改改依赖，试试也没什么损失。要是你从头开始新项目，团队也能hold住Sidecar，那Dapr可能更合适。
