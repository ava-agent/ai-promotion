## 🚨 API Rate Limit 应急处理预案（新增）

### 问题描述
2026-04-03 至 04-11，英文内容生成 Cron 任务 (`8f95598b-a5de-471e-9fbe-c2b88d932449`) 连续触发 `API rate limit reached` 错误，导致内容完全中断9天。

### 影响范围
- 仅影响使用 `kimi-coding/k2p5` 模型的 isolated session 任务
- X平台英文/中文内容生成任务均受此影响
- 每日检查任务本身也多次出现timeout/rate_limit

### 根本根因
isolated session在08:00-09:00时间段执行时，可能与大量其他Kimi专属任务并发（每小时深度分析、趋势追踪、内容审计等），导致API调用频率超过限制。

### 应急修复方案

#### 方案A：切换模型（推荐）
- 英文内容生成任务：从 `kimi-coding/k2p5` 切换为 `zai/glm-5`
- 中文内容生成任务：同步切换为 `zai/glm-5`
- 优势：默认模型更稳定，rate limit阈值更高
- 劣势：创意表达可能略有差异

#### 方案B：调整执行时间
- 英文任务从 08:00 调整至 07:30 或 08:45
- 中文任务从 19:30 调整至 19:00 或 20:15
- 优势：避开Kimi密集任务高峰期
- 劣势：发布时间偏离最佳时段

#### 方案C：增加重试和fallback
- 在任务脚本中增加rate limit检测
- 触发限制时自动等待60秒后重试
- 重试3次失败后切换备用模型
- 优势：无需修改cron配置
- 劣势：实现复杂度较高

### 建议执行
**短期（今天）**: 实施方案A，立即切换模型
**中期（本周）**: 实施方案B，优化时间分布
**长期（本月）**: 实施方案C，建立完整容错机制

### 监控指标
- 连续错误次数 > 2次 → 立即告警
- rate_limit 24小时内出现 > 3次 → 启动应急方案
- 内容生成缺失超过1天 → 自动标记P0告警
