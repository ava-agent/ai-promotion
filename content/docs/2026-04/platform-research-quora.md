## 2. 官方 API 情况

### 2.1 Quora 主站 API

**❌ Quora 没有公开的官方 API。**

- Quora 从未发布过官方开发者 API 用于发帖/回答
- 历史上曾有非官方的 `quora-api` 项目（GitHub 上已标记 UNMAINTAINED/Project discontinued）
- 所有现有工具均为逆向工程，**随时可能失效**
- Quora 使用 GraphQL 内部 API，但未公开文档，且频繁变更

### 2.2 Poe API（Quora 旗下）

**⚠️ Poe 有官方 API，但仅限创建 Bot，不能操作 Quora 内容。**

- Poe（poe.com）是 Quora 旗下的 AI 聊天聚合平台
- `developer.poe.com` 提供官方 API，用于：
  - 创建自定义 AI Bot
  - 接入自己的 LLM 模型
- **不能**通过 Poe API 在 Quora 上发帖、回答问题
- Poe API 与 Quora 主站内容完全独立

### 4.2 推荐工具栈

- **Playwright**（首选，支持反检测插件）
- 配合 `playwright-extra` + `stealth` 插件** 降低被检测概率
- 或使用 `undetected-chromedriver`（Selenium 方案）

### 4.3 登录机制

- **邮箱/密码登录**：最直接，需要处理可能的 2FA
- **Google/Apple SSO**：自动化更复杂，不推荐
- **Cookie 注入**：推荐方案
  - 手动登录后导出 Cookie
  - 自动化时注入 Cookie 绕过登录
  - Quora 使用 `m-b` Cookie 作为主要会话标识

## 5. 反爬措施

### 5.1 Quora 的防护机制

| 措施 | 说明 | 最新发现 | 应对难度 |
|------|------|----------|----------|
| **Cloudflare** | Quora 使用 Cloudflare 保护，有 JS Challenge | 🔴 严重限制自动化访问 | 🟡 中等 |
| **验证码** | 频繁操作触发 reCAPTCHA / hCaptcha | 🔄 验证码频发，难解决 | 🔴 高 |
| **行为检测** | 检测鼠标轨迹、打字速度等 | 🔴 检测精度高 | 🟡 中等 |
| **频率限制** | 单位时间内回答数过多会触发限制 | 🎯 每日≤5条安全 | 🟢 低 |
| **IP 信誉** | 数据中心 IP 容易被封 | 🏠 建议住宅代理 | 🟡 中等 |
| **Headless 检测** | 检测无头浏览器特征 | 🎯 需stealth模式 | 🟡 中等 |

### 5.2 应对策略

- 使用 **residential proxy**（住宅代理）
- Playwright stealth 模式
- 模拟人类行为：随机延迟、鼠标移动、滚动
- 控制频率：每天不超过 5-10 条回答
- 使用已登录的 **user profile** 持久化 Cookie

## 6. 发布格式

- Quora 编辑器为 **富文本编辑器**（基于 contenteditable）
- 支持格式：粗体、斜体、列表、链接、图片、代码块、引用
- **不支持 Markdown 直接输入**（需要通过编辑器按钮转换）
- 建议用 Playwright 操作 DOM 直接插入 HTML 内容
- 图片需要先上传到 Quora（可通过编辑器的上传接口）

## 7. Spam 政策和封号风险

### 7.1 Quora 政策

- **严格禁止**自动化批量操作（违反 Terms of Service）
- **BNBR 政策**（Be Nice, Be Respectful）：内容必须真诚有价值
- **重复内容检测**：相似回答会被折叠或删除
- **新账号限制**：新注册账号发布频率受限
- **回答折叠**：低质量、推广性内容会被折叠（Collapsed）

### 7.2 封号风险等级：⚠️ **高**

- 自动化发帖被检测 → **警告或临时封禁**
- 发布 spam/推广内容 → **永久封禁**
- 同一内容重复发布 → **内容删除 + 账号降权**
- 使用未授权手段 → **违反 ToS，无申诉渠道**

### 7.3 降低风险建议

- 每天回答不超过 **3-5 个**问题
- 回答内容必须**高质量、有差异**（AI 生成需人工审核）
- 选择与账号专业领域匹配的问题
- 间隔时间 **30 分钟以上**
- 养号：先手动使用 1-2 周，建立信任度

## 8. 技术方案推荐

### 方案 A：Playwright 自动化（不推荐）

```
架构：
1. Playwright + stealth 插件
2. Cookie 持久化（登录态保持）
3. 任务队列（控制频率）
4. LLM 生成回答内容 → 人工审核 → 自动发布
```

**难度**：⭐⭐⭐⭐（极高）
**稳定性**：⭐（Quora 前端变更频繁，反爬升级）
**风险**：🔴 高（违反ToS，工具访问受限）
**2026年评估**：**不推荐**，工具验证失败，风险收益比极低

### 方案 B：Quora 内部 API 逆向

```
架构：
1. 抓包分析 Quora GraphQL API
2. 伪造请求头和 Token
3. 直接调用 API 发布
```

**难度**：⭐⭐⭐⭐⭐（极高）
**稳定性**：⭐（API 随时变更，维护成本高）
**风险**：🔴 极高
**2026年评估**：**完全不推荐**，无成熟项目支持，维护成本过高

### 方案 C：半自动（AI 辅助 + 手动发布）

```
架构：
1. 爬取问题列表（手动或简单工具）
2. LLM 生成回答草稿
3. 人工审核确认
4. 手动发布到Quora
```

**难度**：⭐（低）
**风险**：⭐（最低）
**2026年评估**：**推荐**，风险可控，效果稳定

### 方案 D：Poe API + 内容分发（2026年新推荐）

```
架构：
1. 通过Poe API生成AI内容（GPT-4、Claude等）
2. 人工审核优化内容
3. 手动复制发布到Quora和其他平台
4. 可选：开发一键复制粘贴工具
```

**难度**：⭐⭐（低）
**风险**：🟢 低（仅在Poe平台操作）
**2026年评估**：**强烈推荐**，稳定可靠，有成熟API支持
